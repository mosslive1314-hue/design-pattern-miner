## 📋 第一部分：Skill-Architect 设计哲学

### 一泽 Eze 四步循环

#### 1️⃣ 定义成功标准

**技能成功的标准**：
```
1. 用户能在 30 秒内理解任何开源项目的核心思想
2. 提取的内容可直接用于用户自己的项目设计
3. 避免法律风险（提取思想，不复用代码）
4. 支持多种输入来源（GitHub/微信/博客/文档）
5. 输出结构化、可执行、可复用
```

**量化指标**：
| 指标 | 目标值 |
|------|-------|
| 分析时间 | <30 秒 |
| 核心思想提取数 | 3-5 个/项目 |
| 可复用点提取数 | 5-10 个/项目 |
| 用户理解时间 | <10 分钟 |
| 用户满意度 | >90% |

---

#### 2️⃣ 选择起点验证

**MVP 功能**（最小可行产品）：
```
1. 输入：单个 GitHub 项目链接
2. 处理：README + 核心源码分析
3. 输出：设计思想提取报告（Markdown）
```

**验证场景**：
```
输入：https://github.com/AlexAnys/awesome-openclaw-usecases-zh
期望输出：
- 3 个核心设计思想
- 6 个可复用点
- 架构拆解
- 适用场景分析
```

---

#### 3️⃣ 过程校验

**校验点**：
```
1. ✅ 能否正确识别项目类型？
2. ✅ 能否提取核心思想？
3. ✅ 能否抽象出可复用模式？
4. ✅ 输出是否结构化？
5. ✅ 用户能否快速理解？
```

**迭代循环**：
```
设计 → 测试 → 反馈 → 改进 → 再测试
```

---

#### 4️⃣ 对照标准停止

**完成标准**：
```
✅ MVP 功能验证通过
✅ 用户测试满意度 >90%
✅ 提取质量稳定（连续 10 次测试一致）
✅ 文档完整（SKILL.md + README + QUICKSTART）
```

---

### 一泽 Eze 三大设计公式

#### 公式 1: 激发模型能力上限

```
激发模型能力上限 = Agent 策略哲学 + 最小完备工具集 + 必要的事实说明
```

**应用到此技能**：

**Agent 策略哲学**：
```
- 提取思想，不复用代码
- 结构化输出，便于理解
- 场景驱动，非技能驱动
- 难度分级，适合不同用户
```

**最小完备工具集**：
```
1. 网页抓取（GitHub/微信/博客）
2. 语义分析（核心思想提取）
3. 架构分析（模块识别）
4. 模式识别（设计模式）
5. 抽象化（去具体化）
6. 结构化输出（Markdown）
```

**必要的事实说明**：
```
- 授权边界（思想 vs 代码）
- 提取限制（文档不全的项目）
- 使用建议（注明灵感来源）
```

---

#### 公式 2: 技能设计核心

```
好技能 = 清晰的触发条件 + 明确的职责边界 + 可执行的步骤 + 可验证的结果
```

**应用到此技能**：

**清晰的触发条件**：
```
当用户提到：
- "分析这个项目的设计思想"
- "提取实现逻辑"
- "这个是怎么实现的"
- "借鉴这个项目的思路"
- 任何 GitHub/微信/技术文章链接
```

**明确的职责边界**：
```
✅ 做：
- 提取设计思想
- 分析实现逻辑
- 抽象可复用模式
- 生成结构化报告

❌ 不做：
- 直接复制源码
- 安装整个项目
- 深度技术分析（除非用户要求）
- 声称原创
```

**可执行的步骤**：
```
1. 接收 URL
2. 识别来源类型
3. 抓取内容
4. 语义分析
5. 架构拆解
6. 模式抽象
7. 生成报告
```

**可验证的结果**：
```
- 核心思想：3-5 个
- 可复用点：5-10 个
- 架构图：1 个
- 优缺点分析：完整
- 适用场景：明确
```

---

#### 公式 3: 技能触发准确性

```
触发准确性 = 清晰的触发词 + 合适的优先级 + 正确的上下文
```

**触发词设计**：
```
主触发词：
- "分析这个项目"
- "提取设计思想"
- "实现逻辑"
- "架构拆解"

辅助触发：
- GitHub 链接（自动识别）
- 微信文章链接（自动识别）
- 技术博客链接（自动识别）
```

**优先级设置**：
```
priority: 80（中高优先级）
理由：
- 高频使用场景
- 但不覆盖通用搜索技能
```

**上下文处理**：
```
- 记住用户的项目设计目标
- 关联之前分析过的项目
- 支持对比分析（多个项目）
```

## 📋 第二部分：Skill-Creator 工作流程

### 1. 捕获意图

**用户原始需求**：
> "我想设计一个技能，甩给它任何 GitHub 或开源项目或微信文章或任何东西，它可以帮我提取出来整个实现逻辑、抽象的设计思想等，以便我做自己的项目。"

**意图分析**：
```
核心意图：快速学习借鉴他人项目精华
痛点：
- 不想安装整个项目
- 不想逐行阅读源码
- 担心授权问题
- 时间成本高
期望：
- 30 秒理解核心思想
- 提取可复用模式
- 用于自己的项目设计
```

---

### 2. 访谈调研

**关键问题**（需要用户回答）：

#### 问题 1: 输出格式偏好

你希望提取的结果是什么格式？

**选项**：
- [ ] A) 结构化文档（Markdown）- **推荐**
- [ ] B) 流程图 + 文字说明（Mermaid）
- [ ] C) 技能模板（可直接用于 CoPaw）
- [ ] D) 架构 diagrams
- [ ] E) 以上全部 - **推荐**

**建议**：选择 E，因为不同场景需要不同格式

---

#### 问题 2: 提取深度

你希望提取的深度是？

**选项**：
- [ ] A) 概要级（10 分钟理解核心思想）- **推荐**
- [ ] B) 详细级（1 小时理解完整实现）
- [ ] C) 可执行级（可以直接复现）
- [ ] D) 可配置（根据需要选择深度）- **最推荐**

**建议**：选择 D，默认 A，可按需切换

---

#### 问题 3: 输入来源优先级

你最常需要提取的来源是？

**选项**：
- [ ] A) GitHub 项目 - **高频**
- [ ] B) 微信文章 - **高频**
- [ ] C) 技术博客
- [ ] D) 官方文档
- [ ] E) 视频内容（B 站/YouTube）

**建议**：A+B 优先，其他后续支持

---

#### 问题 4: 输出内容优先级

你最关心提取什么？

**选项**：
- [ ] A) 架构设计（模块划分、数据流）- **推荐**
- [ ] B) 核心算法（关键逻辑）
- [ ] C) 实现技巧（代码技巧、优化方法）
- [ ] D) 产品思维（为什么这样设计）- **推荐**
- [ ] E) 可复用组件（可以直接借鉴的部分）- **推荐**

**建议**：A+D+E，因为你要做自己的项目设计

---

#### 问题 5: 授权边界

你对授权的态度是？

**选项**：
- [ ] A) 只提取思想，绝不复用代码 - **推荐**
- [ ] B) 可以复用少量代码（遵守协议）
- [ ] C) 无所谓

**建议**：选择 A，最安全

---

#### 问题 6: 技能名称

你希望技能叫什么？

**候选名称**：
- `design-pattern-miner` - 设计模式挖掘机
- `project-idea-extractor` - 项目思想提取器
- `architecture-analyzer` - 架构分析器
- `idea-miner` - 创意挖掘机
- `pattern-extractor` - 模式提取器

**推荐**：`design-pattern-miner`（清晰表达功能）
